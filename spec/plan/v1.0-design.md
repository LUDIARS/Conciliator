# Conciliator v1.0 設計書 (v0.1 への追補)

> **ステータス**: v1.0 実装済 (typecheck + build + test green、3 モード smoke test 済)
> **更新日**: 2026-05-15
> **前提**: [`spec/requirements.md`](requirements.md) / [`spec/v0.1-design.md`](v0.1-design.md)

v0.1 (単一ホスト loopback) に対し、v1.0 で以下を追加した。

## 1. 動作モード

`conciliator.config.json` の `mode` で 3 モードを切り替える。

| mode | 役割 | DB | watchRoots |
|------|------|----|-----------|
| **standalone** | v0.1 と同じ単一ホスト loopback | あり | 必須 |
| **server** | 中央コーディネータ。遠隔 agent の接続を受ける | あり | 任意 (自身も監視可) |
| **agent** | 各ユーザ PC 常駐。ローカル監視 → server へ転送 | なし | 必須 |

`src/server.ts` の `main()` が mode を見て `runServer()` / `runAgent()` に分岐する。

## 2. マルチホスト + Cernere identity (遠隔ユーザ判定)

### 2.1 構成

```
  ┌─ agent (PC-A) ─┐   ws + Cernere project-token   ┌──────────────────┐
  │ Watcher        ├──────────────────────────────► │ server           │
  │ Cernere 認証    │   (tailnet 経由を想定)          │  /agent gateway  │
  └────────────────┘                                │  → eventBus 再注入 │
  ┌─ agent (PC-B) ─┐                                │  → orchestrator   │
  │ ...            ├──────────────────────────────► │  → engines + DB   │
  └────────────────┘                                │  Web UI (/ws)     │
                                                    └──────────────────┘
```

### 2.2 作業者 identity の解決 (`src/cernere/identity.ts`)

`ResolvedIdentity { host, label, kind, cernereUserId }` に正規化。3 つの由来:

| 由来 | host | cernereUserId | 用途 |
|------|------|---------------|------|
| `identityFromLocal` | この PC | null (or 任意) | standalone の API 宣言 |
| `identityFromLockOwner` | `office-lockfile` | null | Office ロックファイルの owner (未認証) |
| `identityFromAgent` | agent の host | **あり** | agent 経由 = Cernere 認証済み |

`workerIdOf()`: **Cernere user があればそれをキーに** worker id を導出する
(= 同じユーザは host が違っても同一 worker として衝突判定される)。
`workers.cernere_user_id` 列を追加 (schema v2、`migrate()` で既存 DB も追従)。

### 2.3 認証 (`src/cernere/client.ts`)

`feedback_cernere_auth_only_endpoints` に従う:
- **agent 側**: 対話ログイン済みの accessToken (env) → Cernere `/api/auth/project-token`
  (per-user × per-project) で project-token を取得
- **server 側**: agent から受け取った project-token を **ローカル HMAC 検証**
  (`verifyCernereToken` — Cernere は /auth しか公開しないため)
- HMAC secret は環境変数 (`CONCILIATOR_CERNERE_HMAC`、per-user / memory-only)

### 2.4 agent ↔ server プロトコル (`src/agent/protocol.ts`)

WS。agent→server: `agent.hello` (認証) / `agent.snapshot` / `file.changed` /
`lockfile.opened` / `lockfile.closed`。server→agent: `agent.welcome` /
`agent.error` / `notify`。

**スナップショット問題**: ファイル実体は agent 側にしか無い。agent は claim を
起こすイベント (lockfile.opened) の直前に `agent.snapshot` で内容をアップロードし、
server は `stageContent()` で一時保持 → `claimManager.infer()` が `takeStagedContent()`
で baseline に使う。

### 2.5 transport

agent ↔ server は tailnet で閉じる想定 (`feedback_sensitive_data_via_tailscale`)。
identity だけ Cernere、経路は tailnet — Excubitor v0.2 と同じ判断軸。

## 3. AI hook 連携

`tools/conciliator-hook.mjs` — Claude Code 等の PostToolUse(Edit|Write) から発火、
`POST /api/v1/hook/edit` に編集ファイルを通知。監視ルート配下なら **AI 作業者**
として inferred claim を起こす (人間 × AI の衝突も検知対象に)。
opt-in: `CONCILIATOR_HOOK=1`。`.claude/skills/conciliator/SKILL.md` に
衝突通知の解釈ルール (lock しない / ユーザ確認してから編集) を定義。

## 4. Concordia チャット通知チャネル

`src/notify/concordia-channel.ts` — `NotificationChannel` 実装。config に
`concordia` セクションがあれば Notifier のチャネルに追加され、衝突 / リスク等の
通知を Concordia の `POST /v1/chat` に流す。v0.1 で用意した通知レイヤ抽象 (NFR6) の実証。

## 5. クレバーマージの拡充

- **マージ適用** (`src/merge/apply.ts`): `POST /api/v1/collisions/:id/merge` で
  選択したセル解決を書き戻す。surgical な XML 置換で**編集していないセル・スタイル・
  数式は保持**。原本は上書きせず `*.conciliator-merged.xlsx` サイドカーに出力。
- **Maya `.ma` 差分** (`src/merge/maya.ts`): ASCII Maya シーンを `createNode` /
  `setAttr` 解釈で node → attr に展開し、xlsx と同じ 3-way diff (`XlsxDiff` 形) を返す。
  Web UI はノードを「シート」、attr を「セル」として既存ビューアでそのまま表示。
- **`.mb` (バイナリ Maya) は恒久的に非対応** — 専有フォーマットで構造 diff 不能。
