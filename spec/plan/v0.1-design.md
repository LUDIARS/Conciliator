# Conciliator v0.1 設計書

> **ステータス**: v0.1 実装済 (backend / web 実装完了、typecheck + build + test green)

> **対象バージョン**: v0.1

## 1. ミッション

git が苦手な領域 — バイナリ (Excel / Maya / Blender)、意図が分解されていない作業 — で、
**作業の衝突を「起きる前に」検知し、意思確認を挟み、起きてしまった衝突はクレバーに
マージ支援する**ローカル常駐サービス。

Concordia が *AI セッション* を協調させるなら、Conciliator は *ファイルと作業者* を
協調させる。**lock せず**、検知・通知・マージ支援に徹し、作業者の自律解決を促す。

## 2. ラテン語の名前と短縮コード

- **Conciliator** (ラテン: 「和解させる者」「引き合わせる者」)。動詞 *conciliare* (結びつける /
  和解させる) の行為者名詞で、Excubitor / Imperativus と同じ `-or` 系の命名パターンに揃う。
- short code: **Cn** (案 — `LUDIARS/PROJECT-CODES.md` との突き合わせが必要)
- 日本語: コンキリアトル

### 3.1 コンポーネント

| コンポーネント | 役割 | 流用元パターン |
|----------------|------|----------------|
| **Watcher** | 監視ルートを `chokidar` で再帰監視。バイナリ glob にマッチした作成/更新/削除/リネームを `file_event` 化。Office ロックファイル (`~$*.xlsx`) を別途検知して開いている作業者を特定。 | Excubitor `src/catalog/watcher.ts` の debounce ラッパ |
| **Event Bus** | 全コンポーネントが購読する in-process pub/sub。`file.changed` / `claim.created` / `collision.detected` / `structure.violated` / `risk.alerted` 等。 | Concordia `src/events.ts` |
| **Claim Manager** | claim の生成 (明示宣言 / 推測)、baseline スナップショット取得、ライフサイクル (active→stale)、意思確認フローの状態管理。 | — |
| **Collision Engine** | active claim の重複を判定し `collision` 行を起票。事前通知 (FR3.3) と顕在化通知 (FR3.4) を出し分ける。 | — |
| **Structure Checker** | 監視ルートを構成ルールと照合し `structure_violation` を起票。 | Zod ツリースキーマ |
| **Risk Engine** | `file_event` 列にリスクルールを適用し `risk_alert` を起票。 | Excubitor `error_rules` / `error_tasks` の triage モデル |
| **Merge Assistant** | `.xlsx` を ZIP 展開し baseline / A / B をシート・セル単位で diff。競合セルを抽出して構造化 diff を返す。 | — |
| **Notifier** | 通知の出口。v0.1 は WS (Web UI ライブ表示) + OS デスクトップ通知。通知経路は interface 化し、v0.2 で Concordia チャットを足せるようにする。 | — |

### 3.2 設定ファイル `conciliator.config.json`

監視対象・構成ルール・リスクルールの宣言。Custos `apps.json` / Excubitor `services.yaml` と
同じ「宣言ファイルが source of truth」方式。起動時ロード + ファイル watch でホットリロード。

```jsonc
{
  "watchRoots": [
    {
      "id": "design-docs",
      "label": "書類フォルダ",
      "path": "E:/Document/Ars",
      "binaryGlobs": ["**/*.xlsx", "**/*.xlsm", "**/*.mb", "**/*.ma", "**/*.blend"],
      "ignore": ["**/node_modules/**", "**/.git/**"],
      "structure": {
        "requireDirs": ["review"],
        "namingRules": [{ "glob": "**/*.xlsx", "pattern": "^[0-9]{6}_.+\\.xlsx$" }],
        "forbidGlobs": ["**/~$*"]            // ロックファイルは構成違反扱いしない (除外)
      }
    }
  ],
  "riskRules": [
    { "id": "mass-delete",  "kind": "count",   "event": "delete", "windowSec": 60, "threshold": 20 },
    { "id": "secret-leak",  "kind": "content", "match": "(api[_-]?key|secret|-----BEGIN)", "on": "add" },
    { "id": "out-of-scope", "kind": "path",    "match": "outsideWatchRoot" },
    { "id": "perm-change",  "kind": "stat",    "match": "modeChanged" }
  ],
  "claim": {
    "staleAfterSec": 1800,
    "snapshotMaxBytes": 52428800
  }
}
```

## 5. データモデル (SQLite)

```sql
-- 作業者
workers (
  id          TEXT PRIMARY KEY,    -- host + label のハッシュ
  host        TEXT,
  label       TEXT,                -- 人間のユーザ名 / AI セッション role
  kind        TEXT,                -- human | ai
  first_seen  INTEGER,
  last_seen   INTEGER
)

-- 監視ルート (config スナップショット)
watch_roots (
  id          TEXT PRIMARY KEY,
  label       TEXT,
  path        TEXT,
  config_json TEXT,                -- conciliator.config.json の該当エントリ
  loaded_at   INTEGER
)

-- ファイルイベント (TTL 30 日)
file_events (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  ts          INTEGER,
  root_id     TEXT,
  path        TEXT,
  kind        TEXT,                -- add | change | delete | rename | lock-open | lock-close
  worker_id   TEXT,                -- 特定できた場合 (ロックファイル等)
  size_bytes  INTEGER,
  hash        TEXT
)

-- 作業クレーム
claims (
  id            TEXT PRIMARY KEY,
  worker_id     TEXT,
  root_id       TEXT,
  path          TEXT,
  origin        TEXT,              -- declared | inferred
  intent_text   TEXT,              -- 意思確認の回答
  status        TEXT,              -- active | released | stale
  snapshot_id   TEXT,              -- baseline スナップショット
  created_at    INTEGER,
  updated_at    INTEGER,
  released_at   INTEGER
)

-- baseline スナップショット (claim クローズ後 GC 対象)
snapshots (
  id           TEXT PRIMARY KEY,
  claim_id     TEXT,
  path         TEXT,
  hash         TEXT,
  stored_path  TEXT,               -- data/snapshots/ 配下のローカル控え
  size_bytes   INTEGER,
  captured_at  INTEGER
)

-- 衝突
collisions (
  id           TEXT PRIMARY KEY,
  path         TEXT,
  claim_a      TEXT,
  claim_b      TEXT,
  phase        TEXT,               -- pre (事前) | manifest (顕在化)
  status       TEXT,               -- open | ack | resolved | dismissed
  detected_at  INTEGER,
  resolved_at  INTEGER
)

-- 構成ルール / 違反
structure_rules      (id, root_id, rule_json, enabled)
structure_violations (id, root_id, path, rule_id, detail, status, detected_at)

-- リスクルール / アラート
risk_rules  (id, rule_json, enabled)
risk_alerts (id, rule_id, root_id, path, severity, detail, status, detected_at)

-- 監査ログ
audit_log (id, ts, actor, action, target_type, target_id, payload_json)
```

スナップショット実体は `data/snapshots/` にローカル保管。claim が released/stale となり
衝突が resolved/dismissed 済みなら sweeper で GC (個人データ非保管, NFR4)。

## 6. API 概要 (Hono, loopback)

```
GET    /api/v1/watch-roots                  — 監視ルート + 直近イベント数
GET    /api/v1/file-events?root=&since=     — ファイルイベント (cursor)

GET    /api/v1/claims?status=               — claim 一覧
POST   /api/v1/claims                       — 明示宣言で claim 作成
PATCH  /api/v1/claims/:id                   — release / 更新
POST   /api/v1/claims/:id/intent            — 意思確認の回答を記録

GET    /api/v1/collisions?status=           — 衝突一覧
PATCH  /api/v1/collisions/:id               — triage (ack | resolved | dismissed)
GET    /api/v1/collisions/:id/diff          — xlsx セル単位 diff (baseline / A / B)

GET    /api/v1/structure/violations
PATCH  /api/v1/structure/violations/:id     — triage

GET    /api/v1/risk/alerts
PATCH  /api/v1/risk/alerts/:id              — triage
GET    /api/v1/risk/rules
POST   /api/v1/risk/rules                   — ルール追加 / 無効化

GET    /ws                                  — ライブイベント (WS, Web UI 用)
GET    /health
```

## 7. 主要フロー

### 7.1 検知 → 意思確認 → claim

```
1. 作業者 A が facility-budget.xlsx を開く
2. Watcher が ~$facility-budget.xlsx (ロックファイル) の出現を検知
   → ロックファイルを読んで開いているユーザ名 = A を特定
3. Claim Manager: A の inferred claim を生成、baseline スナップショットを取得
4. Notifier: A に意思確認「facility-budget.xlsx で何をしていますか？」
5. A が Web UI で「2026Q1 の予算欄を更新」と回答 → claims.intent_text に記録
```

### 7.2 事前通知 (衝突を起こさせない)

```
1. A の active claim が facility-budget.xlsx に存在
2. 作業者 B が同じファイルを開く → Watcher がロックファイル / mtime で検知
3. Collision Engine: 重複を検知、phase=pre の collision を起票
4. Notifier: B に「A が facility-budget.xlsx を作業中 (意図: 2026Q1 予算欄更新)。
   先に調整してください」をデスクトップ通知 + Web UI で提示
   → B はここで止まれる = 衝突が顕在化しない
```

### 7.3 顕在化した衝突のクレバーマージ

```
1. A と B が両方とも保存してしまった (phase=pre が manifest に昇格)
2. Notifier: 双方に顕在化通知
3. Merge Assistant: baseline / A版 / B版 の 3 つの xlsx を ZIP 展開
   → シート単位、セル単位で比較
   → 「A のみ変更したセル」「B のみ変更したセル」「両者が別の値に変更した競合セル」を分類
4. GET /api/v1/collisions/:id/diff が構造化 diff を返す
5. Web UI が 3-way のセルグリッド diff を表示。人間が見て手作業でマージ (v0.1)
   → v0.2: 選択したセル群を baseline に書き戻す適用支援
```
