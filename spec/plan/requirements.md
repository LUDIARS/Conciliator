# Conciliator 要件整理

> **ステータス**: ドラフト (実装着手前)
> **更新日**: 2026-05-15
> **位置づけ**: v0.1 設計書 (`spec/v0.1-design.md`) の前提となる要件定義

---

## 0. 背景・課題

git をはじめとするテキスト指向のバージョン管理は、以下の領域で**分業がうまく機能しない**:

1. **バイナリファイル** — Excel (`.xlsx` / `.xlsm`)、Maya (`.mb` / `.ma`)、Blender (`.blend`) 等。
   3-way merge が効かず、2 人が同じファイルを更新すれば**確実に衝突**する。後勝ち上書き or
   手作業マージしか手段がない。
2. **意図の分解が不十分な作業** — Maya シーンのように「何を変えたか」がコミット単位に分解
   されておらず、差分を見ても作業者の意図が読み取れない。レビューも統合も困難。
3. **構成のドリフト** — 正しいフォルダ構成 / 命名規約から外れたファイルが混入しても、
   git は「ファイルが増えた」としか教えてくれない。

これらは「コミットしてから衝突に気づく」のでは遅い。**作業を始める前・進行中に検知して
当事者へ知らせ、衝突が起きても賢くマージできる**支援が必要。

Conciliator はこの隙間を埋める。Concordia が *AI セッション* を協調させるのに対し、
Conciliator は *ファイルと作業者* を協調させる。lock はせず、Concordia と同じ
「**助言し、自律解決を促す**」哲学を継ぐ。

---

## 1. スコープ

### 1.1 対象とするもの

- ローカルの**宣言された作業領域** (config で指定したディレクトリ群) の監視
- バイナリ / 非マージファイルに対する**作業の衝突**の検知・予防・マージ支援
- 作業領域の**ファイル構成**が規約どおりかの検証
- ファイル操作に潜む**危険な操作・脆弱性**の検知

### 1.2 対象としないもの (v0.1)

- **Concordia の AI チャット連携** — 将来対応 (v0.2 以降)。v0.1 の通知は Web UI +
  デスクトップ通知に限定する。
- テキストファイルの merge — git の領分。Conciliator は介入しない。
- リモートホスト / 複数 PC 間の協調 — v0.1 は単一ホスト loopback のみ。
- 自動マージの**適用** — v0.1 は構造化 diff の**提示まで**。適用判断は人間が行う。

## 2. 登場概念 (用語)

| 用語 | 定義 |
|------|------|
| **作業領域 (watch root)** | 監視対象として config で宣言したディレクトリ。 |
| **作業者 (worker)** | ファイルを操作する主体。人間 / AI セッションを問わない。host + label で識別。 |
| **作業クレーム (claim)** | 「誰が・どのパスを・何の目的で」触っているかの宣言。明示宣言と検知からの推測の両方で生まれる。 |
| **baseline スナップショット** | claim 開始時点のファイル内容の控え。クレバーマージの 3-way 比較の基点になる。 |
| **衝突 (collision)** | 同一パス (または重複領域) に対して複数の active な claim が存在する状態。 |
| **意思確認 (intent confirmation)** | claim が生まれた / 衝突が検知された作業者に「何をしていますか」を問い、回答を claim に記録するフロー。 |
| **構成ルール (structure rule)** | 作業領域に期待されるフォルダ / 命名構成の宣言。 |
| **リスクルール (risk rule)** | 危険操作・脆弱性を判定するパターン。 |

## 3. 機能要件 (FR)

### FR1. 作業領域の監視

| ID | 要件 |
|----|------|
| FR1.1 | config で宣言された監視ルートを再帰的にファイル監視する。 |
| FR1.2 | バイナリ / 非マージ対象 (glob で宣言) のファイルの作成・更新・削除・リネームを検知する。 |
| FR1.3 | Office のロックファイル (`~$<name>.xlsx` 等) を検知し、**今そのファイルを開いている作業者**を特定する (ロックファイルは開いているユーザ名を保持している)。 |
| FR1.4 | Maya / Blender 等、ネイティブなロックファイルを持たない形式は mtime 変化 + ファイルハンドルの推測で「編集中らしさ」を判定する。 |
| FR1.5 | 監視は debounce し、保存中の中間状態でノイズを出さない。 |

### FR2. 作業クレーム管理

| ID | 要件 |
|----|------|
| FR2.1 | 作業者はパスに対する claim を**明示宣言**できる (Web UI / API)。 |
| FR2.2 | FR1 の検知から claim を**推測生成**できる (ロックファイル出現・更新検知 → 仮 claim)。 |
| FR2.3 | claim 生成時に対象ファイルの baseline スナップショットを取得・保存する。 |
| FR2.4 | claim は active / released / stale のライフサイクルを持つ。一定時間更新が無い claim は stale とする。 |
| FR2.5 | claim には作業者が回答した意図テキスト (FR3.2) を保持する。 |

### FR3. 衝突検知と意思確認

| ID | 要件 |
|----|------|
| FR3.1 | 同一パス (または重複する領域) に複数の active claim が生じたら衝突として記録する。 |
| FR3.2 | claim が生まれた作業者へ「何をしていますか」を問い、回答を claim の意図テキストに記録する。 |
| FR3.3 | **事前通知** — 他者が claim 中のファイルを開いた / 触ろうとした瞬間に、その作業者へ「A が作業中。先に調整を」と通知する。衝突が顕在化する前に止める。 |
| FR3.4 | 衝突が顕在化した場合 (両者が実際に保存した) は、当事者双方へ通知し、クレバーマージ (FR4) の導線を提示する。 |
| FR3.5 | 衝突は ack / resolved / dismissed の triage 状態を持つ。 |

### FR4. クレバーなマージ支援

| ID | 要件 |
|----|------|
| FR4.1 | `.xlsx` / `.xlsm` を ZIP 展開し、シート / セル単位で baseline と各作業者版を比較する。 |
| FR4.2 | 「baseline → A の変更」「baseline → B の変更」を構造化 diff として提示する。競合セル (両者が同一セルを別の値に変更) を明示する。 |
| FR4.3 | v0.1 は**提示まで**。マージ結果の適用は人間が手作業で行う (v0.2 で適用支援)。 |
| FR4.4 | ASCII 形式の Maya シーン (`.ma`) はノード / アトリビュート単位の差分提示を試みる (best-effort、v0.1 はオプション)。 |

### FR5. ファイル構成検証

| ID | 要件 |
|----|------|
| FR5.1 | config で作業領域ごとに期待される構成 (必須ディレクトリ、命名規約 glob、禁止パターン) を宣言できる。 |
| FR5.2 | 監視ルートを構成ルールと照合し、逸脱 (規約外の場所のファイル、命名違反、必須物の欠落) を violation として記録する。 |
| FR5.3 | violation を作業者へ通知する。 |

### FR6. リスク監視

| ID | 要件 |
|----|------|
| FR6.1 | ファイルイベント列にパターンルールを適用し、危険操作を検知する。 |
| FR6.2 | v0.1 で扱う基本リスク: (a) 大量削除・大量リネーム、(b) 秘密情報の混入 (鍵 / トークンらしき内容を含む新規ファイル)、(c) 作業領域外への書き込み、(d) 実行属性 / 権限の変更。 |
| FR6.3 | リスク検知を alert として記録し、作業者へ通知する。 |
| FR6.4 | リスクルールは config / API で追加・無効化できる。 |

### FR7. 通知

| ID | 要件 |
|----|------|
| FR7.1 | v0.1 の通知経路は **Web UI のライブ表示** + **OS デスクトップ通知**。 |
| FR7.2 | 通知対象: 事前通知 (FR3.3)、衝突顕在化 (FR3.4)、構成 violation (FR5.3)、リスク alert (FR6.3)。 |
| FR7.3 | (将来 / v0.2) Concordia チャットへの通知連携。本要件は v0.1 のスコープ外だが、通知レイヤを差し替え可能な抽象として設計する。 |

## 4. 非機能要件 (NFR)

| ID | 要件 |
|----|------|
| NFR1 | **lock しない** — Conciliator はファイルをロックしない。検知・通知・マージ支援のみ。作業者の自律解決を促す。 |
| NFR2 | **LUDIARS スタック準拠** — TypeScript + Node 22 + Hono + better-sqlite3 + Zod、Web は React 19 + Vite + Foundation UI。 |
| NFR3 | **ローカル運用** — loopback (127.0.0.1) bind、v0.1 は認証なし。個人 PC 常駐サービス。 |
| NFR4 | **個人データ非保管** — 作業者の識別は host + label のみ。ファイル内容の baseline スナップショットは衝突解決のためのローカル一時保管とし、claim クローズ後の GC 対象とする (AIFormat §5 準拠)。 |
| NFR5 | **監視の低負荷** — debounce / 対象 glob 限定で、大規模ディレクトリでも CPU / IO を圧迫しない。 |
| NFR6 | **通知レイヤの差し替え可能性** — v0.1 は Web UI + デスクトップ通知だが、Concordia チャット等を後から足せる抽象にする。 |
| NFR7 | **既存資産の流用** — fs 監視 (Excubitor `catalog/watcher.ts`)、ルールエンジン (Excubitor `error_rules` / `error_tasks`)、event bus (Concordia `events.ts`)、設定宣言ファイル (Custos `apps.json` / Excubitor `services.yaml`) のパターンを再利用する。 |

## 5. 受け入れ基準 (v0.1 が「できた」と言える条件)

- [ ] config で宣言した作業領域内の `.xlsx` 更新が検知され、file_event に記録される
- [ ] Excel を開くとロックファイルから「開いている作業者」が特定され、claim が推測生成される
- [ ] claim 生成時に baseline スナップショットが保存される
- [ ] claim の作業者に意思確認が出て、回答が claim に記録される
- [ ] 作業者 A が claim 中のファイルを作業者 B が開くと、B に事前通知が出る
- [ ] A と B が両方保存して衝突が顕在化すると、双方に通知が出る
- [ ] 衝突した `.xlsx` のセル単位 diff (baseline / A / B) が UI に提示される
- [ ] 構成ルールに違反するファイル配置が violation として検知・通知される
- [ ] リスクルール (大量削除 / 秘密混入 / 領域外書込 / 権限変更) が alert として検知・通知される
- [ ] 上記の通知が Web UI のライブ表示 + OS デスクトップ通知の両方で届く

## 6. オープン課題

- 短縮コード (`Cn` 案) の `LUDIARS/PROJECT-CODES.md` との突き合わせ
- ホスト port (17340 / 17341 案) の `infra/PORT-MAP.md` 確定
- Maya `.mb` (バイナリ) の差分提示可否 — v0.1 は `.ma` (ASCII) のみ best-effort
- baseline スナップショットの保管期間と上限サイズの具体値
- 「重複する領域」の判定粒度 — パス完全一致のみか、ディレクトリ単位の重なりも見るか
- デスクトップ通知の実装手段 (OS ネイティブ通知 API / 常駐トレイ / Electron 化の要否)
